OP Labs Case Study · Option 1

Quickstart: from clone to running locally

Get the v1 (walk) platform running on your machine in ~2 minutes, then authenticate your own services right from the running app and generate an executive report.

Before you start

Node ≥ 20 — the platform is TypeScript (ESM).
Docker (optional) — for the real Postgres + pgvector store. Without it, Beacon uses an in-memory store.
git — to clone. No API keys needed to start: GitHub (public) + onchain run with zero keys.

1 · Clone & start

Clone the repo and bring the platform up with a single command. ./beacon up starts Postgres + pgvector (via Docker) and the control plane on it:

git clone https://github.com/alo-code/oplabs.git
cd oplabs
./beacon up                 # → http://localhost:7878

No Docker? Run ./beacon v1 start instead — it uses the in-memory store (everything works; memory just isn't durable across restarts).

Open http://localhost:7878. GitHub and onchain are already live and healthy — no keys. Slack, Notion, and Monday show Connect, because they need a token.

2 · Authenticate your services — right in the app

Two ways — both write to the same gitignored v1-walk/.env and apply immediately:

In the app — click + Connect a service (or a source's Connect), choose the service, paste its token. The connector turns healthy with no restart.
In the terminal — ./beacon setup walks every service (token input is hidden), verifies each, and writes .env.
Slack — a bot token xoxb-… (scope channels:history) + a channel ID. api.slack.com/apps
Notion — an internal integration token secret_…; share your pages with it. notion.so/my-integrations
Monday — an API v2 token + a board ID. developer.monday.com
GitHub (optional) — a read PAT to lift the public rate limit. github.com/settings/tokens
Anthropic (optional) — an API key to upgrade the report to a Claude-written narrative. console.anthropic.com

Where tokens live. Saved to v1-walk/.env (gitignored) and applied to the running process — never committed, never logged (only the variable names appear in logs). The control plane binds to localhost only. This is local-dev convenience over Beacon's central credential registry; in production the same registry is backed by OAuth + a secrets vault (see the Architecture).

3 · Create the report

Click Create Report. Beacon fetches every healthy source into shared memory, refreshes the activity feed, and writes a grounded executive summary — every line links to its real artifact (a GitHub commit, an Etherscan block, a Slack message, a Monday item, a Notion page). With an Anthropic key set, it adds a Claude-written narrative over those same grounded facts.

Just want to see it? Run the demo

No tokens at all — replayed Slack/Notion/Monday data through the real pipeline (labeled “demo data”), alongside live GitHub + onchain:

./beacon scenario               # narrated multi-source runthrough (terminal)
BEACON_FIXTURES=1 ./beacon up   # the control plane with all 5 sources live

Stopping & more

./beacon down                   # stop the control plane AND Postgres
./beacon status                 # is it running? live health
./beacon versions               # what each version is + what's runnable
./beacon v0 demo                # the v0 (crawl) trust gate · ./beacon v0 onchain (a real OP read)

Everything also runs without keys on a fresh clone: cd v1-walk && npm test is green (the suite is offline), and npm run connectors / npm run memory make real, zero-key calls. Full reference: v1-walk/README.md.
