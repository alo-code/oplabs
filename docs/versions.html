Build plan

The four versions — design, stack & architecture

The crawl → walk → run → sprint plan is four directories (v0-crawl → v1-walk → v2-run → v3-sprint), so the line of thinking is browsable. v0 runs today; v1–v3 are design-led, scaffolded code-ready. In the diagrams, solid = built, dashed = design-only.

How to read the four versions — what actually changes

They aren't "small / medium / large." Each answers a different question and removes a different bottleneck — and you only earn the next one with a real-world signal, not a date.

V0 · crawl
“Can we trust one workflow enough to act on it?”
Bottleneck removed: the laptop + blind trust.
V1 · walk
“Can we make the next workflow cheap?”
Bottleneck removed: engineering capacity.
V2 · run
“Can the org run AI work safely without engineers in the loop?”
Bottleneck removed: governance.
V3 · sprint
“Can everyone ask all company knowledge, inline, from any tool?”
Bottleneck removed: reach — the interface itself.

The bottleneck moves — laptop/trust → engineering-capacity → governance → reach (the interface itself). That arc is the plan.

The through-line: trust is the constant

The trust core (grounding + the eval gate) is the spine that survives all four stages — it just graduates form: a Code node in V0 → a shared service in V1 → a governed, audited service in V2 → the boundary of an ambient query in V3 (and, extended to onchain spend policy, the bridge to Option 2). Get the trust machinery right once, on one workflow, then carry it everywhere.

The value in one lineAdding the 2nd workflow (say prospect intel): impossible in V0 (one hardcoded workflow) → days in V1 (reuse connectors + memory + trust) → hours in V2, by a non-engineer. Weeks → days → hours — and in V3, for an answer you don't build a workflow at all; you just ask.

Honest nuanceCrawl doesn't fix all six pains — and that's deliberate. V0 fixes laptop-sleep, observability, non-engineer-readability, and grounding/trust (via n8n + the gate). The re-fetch / shared-memory and central-auth pains are platform-grade — they're V1. The governance layer is V2. The Roadmap has the signals that gate each move.

v0-crawl — the runnable version

Design

The weekly exec brief as an n8n.cloud workflow + a small, tested TypeScript trust core. n8n is the design — it removes three pains for free: always-on, observability (run history), and non-engineer-readable (a visual workflow). The trust core is the only real code, because grounded, measured output is the differentiator.

It pulls a high-signal set — GitHub plus Monday, Notion, one Slack channel, one Drive folder, each just an n8n node — into one grounded brief, then renders two framings: a technical digest for eng leadership and a plain-language “why it matters to customers” brief for BD. Both cite the same PRs, so what BD repeats is safe. The gate runs once; only render and deliver fan out.

v1-walk — the platform (design-only)

Design

Crawl proves one workflow with n8n. Walk is where we outgrow no-code: the 2nd and 3rd workflow shouldn't each re-solve auth, memory, and trust. The one-off becomes reusable infrastructure — a connector library, shared memory, the trust core as a service, a control plane — so the next workflow costs days, not weeks.

v2-run — org-wide & governed (design-only)

Design

Beacon as how OP Labs runs AI work — many workflows, many teams, safely. The platform stops being engineer-gated: non-engineers compose workflows behind guardrails, and the binding constraint shifts from "can we build it" to "who's allowed to and what did it cost" — i.e. governance. This is also the bridge to case-study Option 2: agents that act with value do so through policy-bounded, auditable onchain spend.

v3-sprint — the vision (design-only)

Design

The capstone. After Run, getting an answer still means going somewhere — open the control plane, run a workflow, read the output. Sprint makes the knowledge ambient: every employee asks all of company memory in plain language from the tool they already work in (Claude Code, Cursor, Claude.ai, Slack), through one internal MCP server. It's a thin adapter over the v1/v2 platform — identity in, v2 governance decides, grounding holds. The bottleneck it removes isn't trust, engineering, or governance; it's the interface itself.
