Roadmap — crawl · walk · run · sprint

The first version that earns trust, what comes next, the long-term vision, and the signals that say it's time to move between stages.

The guiding ruleEach stage earns the right to the next by producing a signal, not by hitting a date. We don't build the platform until one workflow has proven it's worth platforming.

CRAWL

One workflow, end to end, trustworthy

Goal: make the weekly exec brief always-on, grounded, and cheap to run — and readable by a non-engineer. Scope: one agent, one real connector (GitHub), shared memory, telemetry, a schedule, a readable render.

Earns trust when: for ~3–4 weeks it runs on schedule with zero silent failures, every figure traces to a source, and a non-engineer reads it unaided.

Move to Walk when:
  • a 2nd team asks "can it do our thing?" (pull, not push)
  • you're copy-pasting connector/memory code to fake a 2nd workflow
  • someone's upset when the brief is late — trust, earned
WALK

A platform, not a script

Goal: turn the one-off into reusable infrastructure so a second and third workflow cost days, not weeks.

  • Connectors as a library (Slack, HubSpot, Notion, Drive, Calendar, onchain) with centralized OAuth/secret management — kill per-source manual auth for good.
  • Shared memory promoted from a file to Postgres + pgvector (structured facts + semantic recall) so agents stop re-fetching and start sharing context.
  • Orchestrator as a real service: scheduling, retries, queues, isolation per run.
  • Observability to a dashboard + alerts; cost/quality SLOs per workflow.
  • Control plane v1: a non-engineer can trigger, schedule, and read any workflow's output and run history from a web UI.
  • Onboard the existing workflows — prospect intel first (the survey ranks it highest-demand and it serves the under-served sales/BD half), then deal decision-log and Q&A.
Move to Run when:
  • non-engineers self-serve workflows without filing a ticket
  • the bottleneck is governance, not engineering ("who's allowed, what did it cost")
  • several teams depend on it daily — an outage would hurt
RUN

Org-wide, self-serve, governed

Goal: Beacon is how OP Labs runs AI work — many workflows, many teams, safely.

  • Self-serve workflow creation for non-engineers (compose existing connectors + prompts behind guardrails) — the platform stops being engineer-gated.
  • Governance: per-workflow budgets, access control, audit log, data-residency + PII policy enforced centrally (the Security/Legal/Compliance partnership the JD calls out).
  • Reliability: SLOs, on-call, graceful degradation, multi-provider model routing with cost/quality routing.
  • Polyglot where it pays: Python agents for eval/ML-heavy work, a Go hot path for high-throughput connectors — behind the same seams, no rewrite.
  • Onchain seam (bridge toward Option 2): agents that act with value do so through policy-bounded, auditable spend — the same trust machinery, extended to money.
Move to Sprint when:
  • non-engineers self-serve daily and the common ask shifts to "can I just ask it?" — not "run me a workflow"
  • people copy workflow output into their own AI tools to ask follow-ups — they want it inline, where they work
  • governance is solid (access control + per-user audit live), so opening a read API org-wide is safe
SPRINT

Ambient — ask the company anything, anywhere

Goal: the platform's knowledge becomes ambient — every employee asks all of company memory in plain language from the tool they already work in (Claude Code, Cursor, Claude.ai, Slack), via one internal MCP server. It's a thin adapter over the v1/v2 platform: identity goes in, v2 governance decides, grounding still holds — it enforces nothing new. Read opens broadly; write/actions stay policy-gated (the same gate that becomes onchain spend — the bridge to Option 2).

Succeeding at Sprint when:
  • daily active queriers far exceed the number of workflows — the median question is answered in-tool in seconds
  • new hires onboard by asking Beacon instead of pinging people; "where do I find…" threads measurably drop
  • AI work is boringly reliable and the live question becomes what to let agents do, not just answer (→ governed onchain action)

Metrics that travel across all four stages

Instrumented from crawl, so the ROI story is real by the time anyone asks — these are the JD's success metrics:

Time saved (hrs/week) and cost (per run + total spend)
Latency and eval/quality score per run
Adoption — active workflows · active non-engineer users
Risk reduction — silent-failure count → 0 · % outputs grounded/audited
