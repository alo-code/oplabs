How we'd build it

Architecture

Beacon is the problem map turned into seams. Every box is a typed contract, so parts are swappable and the system is editable; the shared layers (connectors, memory, orchestrator) are what make it scalable — the 2nd and 3rd workflow reuse them instead of rebuilding them.

A single run, end to end

The control flow is deterministic — the model only summarizes; it never decides what to fetch or whether a claim is grounded. That's what makes the output auditable.

Why it's scalable

The first workflow pays to build the shared layers; every workflow after it rents them. Concretely:

Connectors as a library — one module per source, central auth, zod-parsed, memory-first. A new workflow picks connectors off the shelf; nobody re-solves Slack auth.
Shared memory as a service — agents read/write one store, so context is fetched once and reused. In walk it graduates from a file to Postgres + pgvector behind the same interface.
Orchestrator, not cron-on-a-laptop — scheduling, retries, queues and isolation are solved once for every workflow.
Observability by default — every workflow emits the same run/cost/latency/quality signals, so a 10-workflow platform is as legible as a 1-workflow one.

The test of scalability is the marginal cost of the Nth workflow: with these seams it's days, not weeks, because only the agent logic and prompt are new.

Why it's editable

Workflows are composed, not coded — an agent is connectors + memory + a prompt + an eval. In Run, a non-engineer composes those behind guardrails without touching code.
The model is behind an interface — Claude by default, Gemini/OpenAI swappable per the JD, with a per-call budget guard. No vendor lock-in baked into agents.
Connectors are MCP-shaped — typed tool + schema, so we can reuse existing MCP servers or expose ours to other agents. A Python connector can satisfy the same contract later.
Config-driven — repos, schedules, budgets, thresholds live in config/env, not in code, so behavior is edited without a deploy.

Trust is cross-cutting, not a layer

Three mechanisms wrap every workflow and are the reason a non-engineer can rely on the output: grounding (every claim cites a real artifact — a commit SHA, a PR number), evals (a per-run quality score with a floor that gates publishing), and observability (runs, cost, latency, eval score, and alerts on silent failure). These aren't crawl-or-later polish; they ship in the first slice because trust is the product.
