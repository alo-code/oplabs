How we'd build it

Architecture

Beacon is the problem map turned into seams. Every box is a typed contract, so parts are swappable and the system is editable; the shared layers (connectors, memory, orchestrator) are what make it scalable — the 2nd and 3rd workflow reuse them instead of rebuilding them.

A single run, end to end

The control flow is deterministic — the model only summarizes; it never decides what to fetch or whether a claim is grounded. That's what makes the output auditable.

Why it's scalable

The first workflow pays to build the shared layers; every workflow after it rents them. Concretely:

Connectors as a library — one module per source, central auth, zod-parsed, memory-first. A new workflow picks connectors off the shelf; nobody re-solves Slack auth.
Shared memory as a service — agents read/write one store, so context is fetched once and reused. In walk it graduates from a file to Postgres + pgvector behind the same interface.
Orchestrator, not cron-on-a-laptop — scheduling, retries, queues and isolation are solved once for every workflow.
Observability by default — every workflow emits the same run/cost/latency/quality signals, so a 10-workflow platform is as legible as a 1-workflow one.

The test of scalability is the marginal cost of the Nth workflow: with these seams it's days, not weeks, because only the agent logic and prompt are new.

Why it's editable

Workflows are composed, not coded — an agent is connectors + memory + a prompt + an eval. In Run, a non-engineer composes those behind guardrails without touching code.
The model is behind an interface — Claude by default, Gemini/OpenAI swappable per the JD, with a per-call budget guard. No vendor lock-in baked into agents.
Connectors are MCP-shaped — typed tool + schema, so we can reuse existing MCP servers or expose ours to other agents. A Python connector can satisfy the same contract later. Exposing Beacon itself as an MCP server is v3-sprint.
Config-driven — repos, schedules, budgets, thresholds live in config/env, not in code, so behavior is edited without a deploy.

Trust is cross-cutting, not a layer

Three mechanisms wrap every workflow and are the reason a non-engineer can rely on the output: grounding (every claim cites a real artifact — a commit SHA, a PR number), evals (a per-run quality score with a floor that gates publishing), and observability (runs, cost, latency, eval score, and alerts on silent failure). These aren't crawl-or-later polish; they ship in the first slice because trust is the product.

Who can see what — authorization (v2 → v3)

Grounding, evals, and observability make an answer trustworthy. A fourth mechanism makes it safe to share: authorization — who is allowed to see a given answer. When every employee can ask the whole company in plain language, the trust question stops being “is this grounded?” and becomes “is this person allowed to see what it’s grounded in?” A junior asking “what’s our comp plan?” must not get the board deck. This is arguably the load-bearing piece of trust — it’s how “ask anything” never becomes “read anything.”

The principle: mirror the source’s own permissions — don’t reinvent them. Every fact in shared memory inherits the access list of the artifact it came from: a Slack message is readable by that channel’s members, a Notion page by whoever it’s shared with, a Drive file by its permission list. So authorization isn’t a new policy to maintain — it’s the org’s existing permissions, mirrored onto each memory row. The same question is simply scoped to what you may already see.

Four properties make it trustworthy:

Verified identity, not a claim — the caller arrives as a verified SSO/OIDC identity. The MCP stamps who onto every call and enforces nothing itself; v2’s policy decides — so no one can talk to the MCP to route around governance.
Filtered before retrieval — recall is scoped before the search runs, so out-of-clearance facts never enter the model’s context. You can’t leak what was never retrieved — and the cited set is re-checked live, so a just-revoked access is honored.
Grounding meets clearance — an answer may only cite artifacts you’re allowed to see. If the only grounding is something you can’t see, the honest answer is “no grounded answer available to you” — never a blurred summary that leaks by inference.
Default-deny + per-user audit — anything unlabeled is private, not public; every query is logged per person. That audit trail is also how a mislabel gets caught.

Staged, like everything elsev0/v1 are executive/admin-only, so they need no per-row rules — full read is correct for that audience. v2 introduces the model; v3’s MCP inherits it for free, because the MCP only carries identity in and grounded results out — every decision stays v2’s. The honest hard part is keeping the mirrored permissions fresh as sources change and the labels correct; the per-user audit log is the backstop.
