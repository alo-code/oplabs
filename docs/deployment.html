Deployment & AWS architecture

How the live demo at oplabs-demo.daciasec.net actually runs: the AWS services it uses, how a request is served, how a deploy ships, the identity/trust model — and where each piece goes at production scale. This is an unlinked page (not in the site nav); it documents the real, running system.

ScopeThe live box serves the v1 (walk) control plane — the working slice. v2 / v3 are design-only; their production footprint is the last section (Where it goes). Nothing secret is on this page — no tokens, keys, or hashes; only the architecture and the public resource names.

Runtime topology — how a request is served

  1. Resolve — Cloudflare answers oplabs-demo.daciasec.net with the instance's public IP. The record is DNS-only (grey-cloud) so Caddy's Let's Encrypt HTTP-01 challenge reaches the origin directly.
  2. Terminate + gate — Caddy serves TLS (auto-issued, auto-renewed) and enforces the Basic-Auth perimeter. This is the v1 exec/admin-only boundary made real — authorization at the front door, since v1 has no per-row access control (that's v2).
  3. Proxy — authenticated requests are reverse-proxied to 127.0.0.1:7878. The Node/TS control plane never listens on a public port; only Caddy does.
  4. Serve — the control plane reads from the in-process memory store and drives connectors. GitHub and OP-Mainnet calls are live and keyless; Slack/Notion/Monday return "no token" until connected.

Delivery — how a deploy ships (push-model, no SSH)

  1. Trigger — a push to main touching v1-walk/** (or manual dispatch). Concurrency-guarded so deploys don't overlap.
  2. Package — tar v1-walk, excluding node_modules, .env, logs, and run state — so live tokens on the box are never overwritten.
  3. Authenticate — GitHub's OIDC token is exchanged (via configure-aws-credentials) for short-lived creds on a role trust-scoped to repo:alo-code/oplabs:ref:refs/heads/main. No static secrets exist to leak.
  4. Stage — upload the tarball to the S3 deploy bucket.
  5. Apply — ssm send-command runs a shell script on the instance: pull from S3, extract to /opt/beacon, npm ci as beacon, systemctl restart beacon, then curl the local health endpoint. The job fails unless SSM reports Success and the probe returns HTTP 200.

Identity & trust (IAM)

Two principals, each least-privileged, no long-lived credentials anywhere:

GitHub ActionsOIDC federation to oplabs-demo-gha-deploy, trust policy pinned to the repo + mainwrite the S3 deploy bucket; ssm send-command to the one instance — nothing else
EC2 instanceinstance profile oplabs-demo-ec2-rolebe managed by SSM (core); read the S3 deploy bucket — no write, no broad access
Operator (human)Caddy Basic-Auth at the edge; SSH only from a pinned IP/32 for break-glassreach the gated UI; rarely, shell in to debug

Why it's safe by constructionNo AWS access key lives in the repo or CI. The management plane is SSM, so port 22 isn't needed for deploys (and is pinned to one IP for emergencies). A repo compromise can't assume the deploy role unless it's on main.

Where it goes — production scale

The crawl-walk-run discipline applies to the infrastructure itself. Each current piece has a clean production replacement behind the same seam, so the move is config + IaC, not a rewrite. The drivers are exactly the v2/v3 needs from the architecture: shared queryable memory, per-user identity, and governed access.

SeamToday (crawl/walk)Production (run/sprint)Driver
Computeone EC2 box, systemdECS/Fargate (or EKS) behind an ALB, multi-AZreliability, SLOs, horizontal scale
Memoryin-process storeRDS Postgres + pgvector (same MemoryStore interface)shared, durable, queryable across hosts — and the home of the ACL-scoped rows
Secrets.env on hostSecrets Manager / SSM Parameter Store + rotationcentral rotation/audit; no secrets on disk
PerimeterCaddy Basic-AuthSSO / OIDC IdP (e.g. Cognito) → per-user identityv2/v3 access control needs a verified caller, not a shared password
TLS / edgeCaddy + Let's EncryptACM + ALB (or keep Caddy)managed certs, health checks, WAF
DeliveryGitHub Actions → S3 → SSMsame, + IaC (Terraform/CDK), staged envs, blue/greenrepeatable, reviewable, reversible infra
Observabilitystructured logs + OTLP hooksCloudWatch / OTel collector, alerts on silent failureruns, cost, latency, eval score, grounding %

The through-lineNone of these is a rewrite. The app depends on interfaces (a memory store, a credential registry, a connector contract), so "EC2 → Fargate" and "in-memory → RDS" are swaps behind seams that already exist. That's the same reason the v2 access-control model lands as a config flip, not a re-architecture.
