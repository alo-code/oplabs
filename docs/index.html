OP Labs Case Study · Option 1

Beacon: an always-on agent platform for internal AI workflows

The approach document. Problem → solution → why → stakeholders → plan, written for a non-technical reader. The pages link together, top-right.

The problem

OP Labs runs a growing set of AI workflows over company data in Slack, HubSpot, Notion, Drive, Calendar, GitHub, and onchain sources — weekly exec briefs, a deal decision-log, prospect intel, a Q&A agent. They produce real value, but they're stitched together by one person on a laptop and they don't scale:

🔁 Context is re-fetched every task — slow and expensive; cost grows per workflow.
😴 It sleeps when the laptop sleeps — no schedule, no reliability.
🔑 Auth is per-source and manual — brittle, and a security blind spot.
🧠 No shared memory — agents can't build on each other's context.
🔭 No observability — failures are silent; nobody knows the cost.
🙋 Can't hand it to a non-engineer — one person is the bottleneck and the risk.

See the full mapping of each pain to its fix on the Problem map.

The proposed solution

Beacon is a thin, always-on platform over OP Labs' existing data sources that runs the AI workflows as managed services — with shared memory, centralized auth, built-in observability, and a control plane a non-engineer can use. Trust is a feature: every output is grounded in a real source and every run is measured, so the org can believe what comes out. Detail on the Architecture page.

Why this, over the alternatives

AlternativeWhy not (alone)
Keep the laptop scriptsSingle point of failure, no trust mechanism, cost compounds with every new workflow. This is the status quo we were asked to fix.
Buy an off-the-shelf agent platformOur data lives across private OP systems + onchain; generic tools don't ground in our sources or meet our security/governance bar, and add lock-in. We reuse open standards (MCP, model SDKs) — we don't outsource the trust layer.
Let each team build its ownDuplicated auth, memory, and observability; inconsistent (and unauditable) trust; no shared context across agents.

Beacon is the smallest thing that removes the six pains and earns trust — built incrementally so it proves value before it asks for investment.

Stakeholders & how I'd gather requirements

OP Labs is ~75 people, roughly half in sales/BD and half engineers. That split matters: the sales/BD half are the heaviest consumers of the AI output (prospect intel, deal log) yet — being non-engineers — are exactly the people locked out of self-serving it today. They're the largest under-served audience.

Sales / BD / GTM (~37) — consume prospect intel + deal log, can't self-serve. Want it before every call; trust = sources they can click, never stale.
Engineering (~37) — the builders; one of them is the laptop. Want reusable connectors/memory + loud failures; trust = it runs without them.
Leadership / Exec — read the brief, own buy-in + budget. Trust = on time, grounded enough to defend a decision.
Security / Legal / Compliance — privacy, secrets, audit. Bring them in at crawl; they gate the self-serve (Run) stage.

Method — gather, then keep gathering. Rather than guess, I'd survey all 75 (two tracks: consumers vs. builders) and score workflows by reach × frequency × pain ÷ effort, so the roadmap is pull-driven. A short pulse re-runs at each stage gate — requirements keep shaping the plan, and it doubles as an adoption metric. Demand sets build order; a trust-risk filter sets what ships first — which is why crawl is the exec brief, even though raw demand ranks prospect intel higher.

The plan, in one line

Crawl: make one workflow (the weekly exec brief) always-on, grounded, and trusted. Walk: turn the one-off into reusable connectors + memory + control plane so the 2nd and 3rd workflows cost days. Run: org-wide, self-serve, governed. Each stage earns the next via a signal, not a date — see the Roadmap.

A note on language (TypeScript)

The JD lists Python and Go; this prototype is TypeScript — a deliberate call. The grade is judgment and orchestration, and "it has to run," so my strongest language gives the cleanest slice. The agent/MCP/onchain tooling (Vercel AI SDK, MCP TS SDK, viem) is first-class in TS, and end-to-end type safety removes a bug class — which matters when handing to a non-engineer. The production vision stays polyglot (Python for eval/ML agents, a Go hot path where needed) behind language-agnostic seams: tool schemas, a bus, HTTP.
