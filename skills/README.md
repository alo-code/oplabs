# skills/ — how we build with AI here

These are the playbooks an engineer (or an AI agent) reads before doing a
piece of work in Beacon. They make the *method* explicit so the work is
consistent whoever — or whatever — is at the keyboard.

They are not generic best-practice essays. Each one is a short, opinionated
procedure tied to this repo's cadence and architecture.

| Skill | Read it when |
|---|---|
| [`ai-driven-development.md`](./ai-driven-development.md) | Starting any task — how we drive + review AI here |
| [`cadence-tdd.md`](./cadence-tdd.md) | Writing any slice (contract → red → observability → green → evidence) |
| [`writing-workplans.md`](./writing-workplans.md) | Turning a problem into a sliced workplan in `todo/` |
| [`connector-pattern.md`](./connector-pattern.md) | Adding a data source (Slack, GitHub, Notion, onchain…) |
| [`agent-pattern.md`](./agent-pattern.md) | Adding a workflow/agent over connectors + memory |
| [`evals-and-observability.md`](./evals-and-observability.md) | Making agent output testable + runs observable |
| [`working-with-notes.md`](./working-with-notes.md) | Every session — keeping `notes.md` honest |
| [`docs-site.md`](./docs-site.md) | Building or editing the `docs/` approach site (pages + SVG diagrams) |
| [`n8n-workflow.md`](./n8n-workflow.md) | Changing the v0 n8n workflow (or editing it as a non-engineer) |

> If you find yourself doing something repeatedly that isn't written down,
> that's a missing skill. Add it here in the same shape.
